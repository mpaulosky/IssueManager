## Decision: Standardized Handler Test Patterns

### Pattern Overview

All handler tests follow a consistent 6-pattern structure:

1. **Happy path:** Valid input returns expected result
2. **Validation failures:** Empty fields, too long, too short, invalid formats
3. **Not found scenarios:** Returns null or throws NotFoundException
4. **Repository failures:** Throws appropriate exceptions with error messages
5. **Idempotent operations:** Repeated calls (e.g., archiving archived entity) succeed without side effects
6. **Cancellation token:** Token is passed correctly to repository methods

## Entity-Specific Patterns

### Issue Handlers (Direct DTO Pattern)

**Key difference:** IssueRepository returns `IssueDto` directly (no Result<T> wrapper).

```csharp
// Repository returns DTO directly
_repository.GetByIdAsync(issueId, Arg.Any<CancellationToken>())
    .Returns(issueDto);

// Handler returns DTO directly
var result = await _handler.Handle(query, CancellationToken.None);
result.Should().NotBeNull();
result!.Title.Should().Be("Expected Title");
```

**Example test:**

```csharp
[Fact]
public async Task Handle_ValidIssueId_ReturnsIssue()
{
    // Arrange
    var issueId = ObjectId.GenerateNewId().ToString();
    var expectedIssue = IssueBuilder.Default()
        .WithId(issueId)
        .WithTitle("Test Issue")
        .Build();

    _repository.GetByIdAsync(issueId, Arg.Any<CancellationToken>())
        .Returns(expectedIssue);

    var query = new GetIssueQuery(issueId);

    // Act
    var result = await _handler.Handle(query, CancellationToken.None);

    // Assert
    result.Should().NotBeNull();
    result!.Title.Should().Be("Test Issue");
}
```

### Category/Status/Comment Handlers (Result<T> Pattern)

**Key difference:** Repositories return `Result<T>` (where T = model), handlers map to DTO.

```csharp
// Repository returns Result<Model>
_repository.GetAsync(categoryId)
    .Returns(Result<Category>.Ok(category));

// Handler maps model.ToDto()
var result = await _handler.Handle(query, CancellationToken.None);
result.Should().NotBeNull();
result!.CategoryName.Should().Be("Bug");
```

**Example test:**

```csharp
[Fact]
public async Task Handle_ValidCategoryId_ReturnsCategory()
{
    // Arrange
    var categoryId = ObjectId.GenerateNewId();
    var category = new Category
    {
        Id = categoryId,
        CategoryName = "Bug",
        CategoryDescription = "Bug reports"
    };

    _repository.GetAsync(categoryId)
        .Returns(Result<Category>.Ok(category));

    var query = new GetCategoryQuery(categoryId.ToString());

    // Act
    var result = await _handler.Handle(query, CancellationToken.None);

    // Assert
    result.Should().NotBeNull();
    result!.CategoryName.Should().Be("Bug");
}
```

## Common Test Scenarios

### 1. Create Handler Tests

| Scenario | Expected Outcome |
|----------|------------------|
| Valid input | Returns created entity DTO |
| Empty name | Throws ValidationException("*name*required*") |
| Name too short | Throws ValidationException("*at least X characters*") |
| Name too long | Throws ValidationException("*exceed X characters*") |
| Description too long | Throws ValidationException("*exceed X characters*") |
| Null description | Uses empty string (no error) |
| Repository failure | Throws InvalidOperationException(error message) |
| Cancellation token | Token passed to repository |

### 2. Get Handler Tests

| Scenario | Expected Outcome |
|----------|------------------|
| Valid ID | Returns entity DTO |
| Non-existent ID | Returns null |
| Empty ID | Throws ArgumentException("*ID cannot be empty*") |
| Whitespace ID | Throws ArgumentException("*ID cannot be empty*") |
| Invalid ObjectId | Returns null (no repository call) |
| Cancellation token | Token passed to repository |

### 3. Delete Handler Tests

| Scenario | Expected Outcome |
|----------|------------------|
| Valid entity | Returns true, archives entity |
| Non-existent entity | Throws NotFoundException("*ID*was not found*") |
| Already archived | Returns true (idempotent, no archive call) |
| Empty ID | Throws ValidationException("*ID*required*") |
| Invalid ObjectId | Throws NotFoundException |
| Cancellation token | Token passed to repository |

### 4. List Handler Tests

| Scenario | Expected Outcome |
|----------|------------------|
| Entities exist | Returns all entities as DTOs |
| No entities | Returns empty list |
| Repository failure | Returns empty list |
| Repository returns null | Returns empty list |
| Cancellation token | Token passed to repository |

### 5. Update Handler Tests

| Scenario | Expected Outcome |
|----------|------------------|
| Valid input | Returns updated entity DTO |
| Empty ID | Throws ValidationException("*ID*required*") |

## Builder Pattern Implementation

### Why Builders?

Builders provide:
- **Fluent API** for readable test data creation
- **Sensible defaults** (no need to specify all fields)
- **Reusability** across test classes
- **Type safety** via method chaining

### Example: CategoryBuilder

```csharp
public class CategoryBuilder
{
    private ObjectId _id = ObjectId.GenerateNewId();
    private string _categoryName = "Default Category";
    private string _categoryDescription = "Default category description";
    private DateTime _dateCreated = DateTime.UtcNow;
    private DateTime? _dateModified;

    public CategoryBuilder WithId(string id)
    {
        _id = ObjectId.Parse(id);
        return this;
    }

    public CategoryBuilder WithCategoryName(string categoryName)
    {
        _categoryName = categoryName;
        return this;
    }

    public CategoryDto Build() =>
        new(
            _id,
            _categoryName,
            _categoryDescription,
            _dateCreated,
            _dateModified);

    public static CategoryBuilder Default() => new();
}
```

### Usage in Tests

```csharp
// Default values
var category = CategoryBuilder.Default().Build();

// Custom values with fluent API
var customCategory = CategoryBuilder.Default()
    .WithCategoryName("Bug")
    .WithCategoryDescription("Bug reports")
    .Build();

// Chaining multiple setters
var complexCategory = CategoryBuilder.Default()
    .WithId(categoryId.ToString())
    .WithCategoryName("Feature Request")
    .WithCategoryDescription("User feature requests")
    .WithCreatedAt(DateTime.UtcNow.AddDays(-7))
    .Build();
```

## Lessons Learned

### 1. Result<T> vs. Direct DTO Patterns

**Discovery:** Issue handlers use direct DTO returns, while Category/Status/Comment use Result<T>.

**Impact on tests:**
- Issue tests mock repository returning `IssueDto` directly
- Other tests mock repository returning `Result<Category>.Ok(model)`
- Handlers for non-Issue entities must call `.ToDto()` on model

**Recommendation:** Consider standardizing on Result<T> pattern across all handlers for consistency.

### 2. Validation Rule Discrepancies

**Discovery:** Validators use different limits than expected:
- Issue.Title max is **200** (not 256)
- Issue.Description max is **5000** (not 4096)

**Lesson:** Always check actual validator rules before writing tests. Do not assume standard limits.

### 3. ObjectId Validation

**Discovery:** Category/Status/Comment handlers perform `ObjectId.TryParse()` checks before repository calls.

**Pattern established:**
```csharp
if (!MongoDB.Bson.ObjectId.TryParse(query.CategoryId, out var objectId))
    return null; // Early exit, no repository call
```

**Test implication:** Must test invalid ObjectId returns null **without** calling repository.

### 4. Idempotent Delete Operations

**Discovery:** All delete handlers check `entity.Archived` and return true if already archived.

**Pattern established:**
```csharp
if (category.Archived)
    return true; // No archive call needed
```

**Test implication:** Must test "already archived" scenario does **not** call `ArchiveAsync()`.

## Recommendations for Future Handler Development

1. **Use the established patterns** — all new handlers should follow the 6-pattern structure
2. **Write tests first** (TDD) — validator rules, error messages, and edge cases are discovered faster
3. **Run tests locally** before pushing — zero warnings/errors enforced
4. **Use builders for complex DTOs** — reduces test maintenance burden
5. **Standardize repository patterns** — consider migrating Issue handlers to Result<T> pattern
6. **Document validation rules** — centralize min/max limits for reference
