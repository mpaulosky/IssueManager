## Branching Strategy Audit — IssueManager

### Current State

**Active branches (by recency):**

| Branch | Behind main | Ahead of main | Files changed | Status |
|--------|------------|---------------|---------------|--------|
| `squad/34-tailwindcss-ui-redesign` | 0 | 9 | 1 | Open PR #36 |
| `squad/fix-dto-architecture` | 1 | 10 | 118 | No open PR |
| `feature/adding-new-items` | 3 | 0 | 0 (merged) | Stale |

**Key findings:**

1. **Both active squad branches forked from the same base** (commit `d29593f`, Feb 17) and have been alive for ~6 days.
2. **`fix-dto-architecture` touches 118 files** — a massive refactor spanning handlers, DTOs, mappers, validators, tests, workflows, and squad metadata. This is the conflict amplifier.
3. **`34-tailwindcss-ui-redesign` touches only 1 source file** (`src/Api/Program.cs`) but that file is also modified by `fix-dto-architecture`.
4. **Both branches already have "resolve merge conflicts" commits** (`019cdff`, `ff87c3a`), confirming repeated conflict resolution.
5. **Branch protection is NOT enabled on `main`** — no required reviews, no status checks, no "require branch to be up to date" rule.
6. **No automated sync mechanism exists** — no workflow rebases or auto-updates squad branches when main changes.
7. **`.gitattributes` exists** with `merge=union` for squad state files, which is good but doesn't cover source code.

### Root Cause

**Primary: A 118-file architectural refactor (`fix-dto-architecture`) running concurrently with feature work, with no branch protection or sync discipline.**

Contributing factors (ordered by impact):

1. **Oversized branch scope** — `fix-dto-architecture` changes namespace structure, DTOs, mappers, handlers, validators, and tests all in one branch. Any other branch touching the API layer will conflict.
2. **No "require up to date" branch protection** — PRs can be merged without being current with main, causing the next PR to inherit all conflicts.
3. **No automated rebase/sync** — branches drift silently. Conflicts are discovered only at merge time.
4. **PR merge ordering not coordinated** — the architectural refactor (`fix-dto-architecture`) was not merged first, so `34-tailwindcss-ui-redesign` had to manually resolve conflicts with code it didn't change.
5. **Squad metadata files (.squad/)** in branches — 60+ `.squad/` files appear in diffs, inflating change counts and creating noise conflicts.

### Hot Files

Files changed by multiple concurrent branches (conflict magnets):

| File | Branches touching it | Conflict risk |
|------|---------------------|---------------|
| `src/Api/Program.cs` | tailwindcss, fix-dto | **HIGH** — both change `using` directives and endpoint signatures |

Potential future hot files if more branches open:

- `src/Shared/DTOs/IssueDto.cs` — central DTO used everywhere
- `tests/Unit/Handlers/*.cs` — handler tests change with any handler refactor
- `src/Api/Handlers/Issues/*.cs` — handler implementations
- `.github/workflows/squad-ci.yml` — CI config modified by infrastructure branches

### Recommended Process Changes

#### Immediate (manual)

1. **Merge architectural refactors FIRST.** `fix-dto-architecture` must be merged before any feature branch. It touches 118 files — every other branch will conflict with it. Create a PR for it immediately and merge it today.
2. **Rebase `34-tailwindcss-ui-redesign` onto main AFTER the DTO refactor merges.** This will cleanly resolve its single-file conflict.
3. **Sync squad branches with main daily.** Every squad member should run `git fetch origin && git rebase origin/main` at the start of each session.
4. **Keep `.squad/` files out of PR branches.** Add `.squad/` to `.gitignore` on feature/squad branches, or only commit squad files on dedicated metadata branches.

#### Automated Safeguards

1. **Enable branch protection on `main`:**
   - Require pull request reviews (at least 1)
   - Require status checks to pass
   - **Require branches to be up to date before merging** — this is the single most impactful setting
   - Enforce for administrators

2. **Add an auto-sync workflow** (`.github/workflows/squad-sync.yml`):

   ```yaml
   name: Sync squad branches with main
   on:
     push:
       branches: [main]
   jobs:
     sync:
       runs-on: ubuntu-latest
       strategy:
         matrix:
           branch: # dynamically list squad/* branches
       steps:
         - uses: actions/checkout@v4
           with:
             ref: ${{ matrix.branch }}
             fetch-depth: 0
         - run: |
             git merge origin/main --no-edit || echo "::warning::Merge conflict on ${{ matrix.branch }}"
             git push
   ```

3. **Add `.gitattributes` merge strategies** for known hot files:

   ```
   # Auto-merge using directives (order doesn't matter)
   src/Api/Program.cs merge=union
   ```

   Note: `merge=union` is only safe for files where line order is irrelevant (like `using` blocks). Use with caution.

#### Team Workflow Rules

1. **Maximum branch lifetime: 3 days.** If a branch can't be merged in 3 days, it's too large. Split it.
2. **Maximum files per PR: 30.** The 118-file `fix-dto-architecture` should have been 4-5 separate PRs:
   - PR1: Rename namespaces
   - PR2: Add `Shared.Mappers`
   - PR3: Migrate handlers to `IssueDto`
   - PR4: Update tests
   - PR5: Update workflows
3. **Merge order protocol:** When multiple PRs are in flight, merge in this order:
   - Infrastructure/CI changes first
   - Architectural refactors second
   - Feature work last
4. **No concurrent refactors.** Only one namespace/DTO/architecture refactor branch may be active at a time. Other squad members work on non-overlapping areas or wait.
5. **Squad metadata isolation.** `.squad/` changes should be committed to a dedicated `squad/metadata` branch or kept out of source branches entirely.

### Priority Order

| Priority | Action | Impact | Effort |
|----------|--------|--------|--------|
| 🔴 P0 | Merge `fix-dto-architecture` PR first | Eliminates 118-file conflict source | Low |
| 🔴 P0 | Enable "require up to date" branch protection | Prevents stale merges | 5 min |
| 🟡 P1 | Rebase `34-tailwindcss-ui-redesign` after DTO merge | Clean conflict resolution | Low |
| 🟡 P1 | Establish 30-file PR size limit | Prevents future mega-branches | Process |
| 🟢 P2 | Add auto-sync workflow | Catches drift early | Medium |
| 🟢 P2 | Isolate `.squad/` files from source branches | Reduces noise conflicts | Low |
| 🔵 P3 | Add `.gitattributes` merge strategies | Handles trivial conflicts | Low |

### Decision

**Approved process rules effective immediately:**

1. Branch protection with "require up to date" MUST be enabled on `main`.
2. Architectural refactors merge before feature work — no exceptions.
3. PR size limit: 30 files maximum. Larger changes must be split.
4. Squad branches sync with main daily via rebase.
5. Maximum branch lifetime: 3 working days.

## Recommended Approach: GitHub Native Auto-Generated Release Notes

### What It Does
GitHub's native feature automatically categorizes merged PRs into release notes sections based on:
- PR labels (`type:feature`, `type:bug`, `type:chore`, etc.)
- PR titles and descriptions
- Contributor associations

### Configuration Files Required

#### 1. `.github/release.yml` (NEW)
Create this file at the repository root:

```yaml
# GitHub Auto-Generated Release Notes Configuration
# Docs: https://docs.github.com/en/repositories/releasing-projects-on-github/automatically-generated-release-notes

changelog:
  exclude:
    labels:
      - duplicate
      - invalid
      - wontfix
      - question
    authors:
      - dependabot
  categories:
    - title: 🎉 Major Features & Enhancements
      labels:
        - type:feature
        - enhancement
    - title: 🔧 Improvements
      labels:
        - type:chore
        - refactor
    - title: 📚 Documentation
      labels:
        - type:docs
        - documentation
    - title: 🐛 Bug Fixes
      labels:
        - type:bug
        - bug
    - title: 🔬 Research & Spikes
      labels:
        - type:spike
    - title: 🚀 Infrastructure & DevOps
      labels:
        - priority:p0
```

#### 2. Update `.github/workflows/squad-release.yml` (MODIFIED)
Replace the TODO placeholder with actual release logic:

```yaml
name: Squad Release

on:
  push:
    branches: [main]

permissions:
  contents: write

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v6
        with:
          fetch-depth: 0

      - name: Set up .NET
        uses: actions/setup-dotnet@v4
        with:
          dotnet-version: '10.0.100'

      - name: Build and test
        run: dotnet test --configuration Release --no-restore

      - name: Get version from git tag
        id: tag
        run: |
          VERSION=$(git describe --tags --abbrev=0 2>/dev/null || echo "v0.0.0")
          echo "version=$VERSION" >> $GITHUB_OUTPUT

      - name: Check if version is new (not already released)
        id: check_release
        run: |
          VERSION=$(git describe --tags --abbrev=0 2>/dev/null || echo "v0.0.0")
          RELEASE_EXISTS=$(gh release view "$VERSION" 2>/dev/null && echo "true" || echo "false")
          echo "release_exists=$RELEASE_EXISTS" >> $GITHUB_OUTPUT
        env:
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}

      - name: Create release with auto-generated notes
        if: steps.check_release.outputs.release_exists == 'false'
        run: |
          VERSION=$(git describe --tags --abbrev=0 2>/dev/null || echo "v0.0.0")
          gh release create "$VERSION" \
            --title "$VERSION" \
            --generate-notes \
            --repo ${{ github.repository }}
        env:
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}

      - name: Release already exists
        if: steps.check_release.outputs.release_exists == 'true'
        run: echo "✅ Release ${{ steps.tag.outputs.version }} already published"
```

## How It Works: The Flow

### Current Flow (Status Quo)
```
dev ────────────────────┐
                        ├─→ preview ──────┐
                                          ├─→ main (squad-release.yml runs)
                                          │   → creates empty release
                                          └─→ tag created manually
```

### Recommended Flow (With Release Notes)
```
1. PR merged to main (with labels: type:feature, type:bug, etc.)
   ↓
2. Git tag pushed to main (v0.0.x)
   ↓
3. squad-release.yml triggered
   ├─ Runs dotnet test
   ├─ Detects new tag (not already released)
   └─ Creates release with AUTO-GENERATED notes
      ├─ Categorizes PRs by label
      ├─ Groups into sections (Features, Bugs, Chores, etc.)
      ├─ Links to contributors
      └─ Embeds milestone info (if PR references a milestone)
```
